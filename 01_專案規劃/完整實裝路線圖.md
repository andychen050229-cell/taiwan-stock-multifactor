## 三、Phase 2 實裝計畫（模型核心）

### 3.1 建立 Nested Walk-Forward 分割器
**複雜度**: M | **優先級**: CRITICAL | **預計工期**: 2-3 週

#### 3.1.1 實現 Purged Walk-Forward 分割
- [ ] 建立 `src/backtest/splitter.py`
  - 複雜度: M
  - 概念:
    - 參數: train_window=504 日, test_window=63 日, step=63 日, embargo=20 日
    - 避免 lookahead bias: test 日期前 embargo 日內資料不進 train
    - Purged (去重): 如有重疊樣本，從 train 中移除
  - 實現步驟:
    1. 產生時間戳清單
    2. 逐次滑動 train/test 視窗
    3. embargo 邏輯: train_end = test_start - embargo_days
    4. 抽樣 purge: 移除 test 中出現過的 (ticker, date) pair
  - 輸出: list of (train_idx, test_idx) tuple
  - 驗證: 確保 train/test 無重疊、無 lookahead

#### 3.1.2 三層 Nested 結構
- [ ] 支持 nested cross-validation
  - 複雜度: S

### 3.7 樣本外穩健性評估（OOD Stress Testing）
**複雜度**: M | **優先級**: HIGH | **預計工期**: 2 週

#### 3.7.1 設計五大壓力情景
- [ ] 建立 `src/backtest/stress_tests.py`
  - 複雜度: M
  - 情景 1: 時間漂移（Temporal drift）
    - 測試: 對最新 252 日（1 年）資料，所有模型 AUC 是否下降 > 10%？
    - 如是 → 警告：模型已過時
  - 情景 2: 波動爆漲
    - 定義: 歷史波動率 > 75 百分位
    - 測試: 高波動期間模型 AUC 是否顯著下降？
  - 情景 3: 市場利空（跌幅 > 10%）
    - 定義: 5 日市場報酬 < -10%
    - 測試: 下跌市況中 short signal precision 是否高？
  - 情景 4: 股災區間
    - 定義: max drawdown > 20% 的期間
    - 測試: 模型是否能迴避最深回撤？
  - 情景 5: 流動性枯竭
    - 定義: volume < 10 百分位
    - 測試: 低流動性股票模型是否失準？
  - 輸出: 5 個情景 × 18 模型 = 90 個 AUC 指標

#### 3.7.2 Kill Criteria（擊殺規則）
- [ ] 定義回測前的淘汰標準
  - 複雜度: S
  - 規則 1: 任何情景下 AUC < 0.50 → KILL
  - 規則 2: 時間漂移下降 > 20% → KILL
  - 規則 3: short signal precision < 0.40 → KILL
  - 規則 4: feature importance 集中在 1-2 特徵 → WARNING
  - 輸出: Surviving models list

## 八、風險與緩解計畫

| 風險 | 發生機率 | 影響 | 緩解策略 |
|------|---------|------|---------|
| OHLCV 資料無法取得 | HIGH | 特徵工程延遲 | 提前決定備用來源；先用 closing_price 推導替代指標 |
| 標籤嚴重不平衡 | MEDIUM | 模型偏差 | 使用 class_weight 或 SMOTE；評估 AUC 而非準度 |
| 超參數搜索耗時超長 | MEDIUM | 研發進度延遲 | 限制 n_trials；使用 GPU；早期停止 |
| 所有模型 AUC < 0.52 | MEDIUM | Phase 2 失敗 | 檢查標籤定義；增加特徵；回到特徵工程 |
| 模型在 OOD 測試下崩潰 | MEDIUM | 部署風險 | 提前設計 Kill criteria；增加風險特徵權重 |
| UI 開發時間超期 | LOW | Phase 3 延遲 | 優先做資料表格版，再做圖表版 |

## 九、檢查點與評估準則

### Phase 1 完成評估準則
- [ ] 特徵工程: 所有五支柱各 ≥ 10 個特徵
- [ ] 去重後特徵: 總數 50-100（避免過多或過少）
- [ ] 標籤平衡: 各 horizon 的正類比例在 30%-70% 範圍內
- [ ] 缺值: 所有特徵缺值率 < 30%
- [ ] 相關性: max feature correlation < 0.95
- [ ] 品質閘門: 4/4 通過（ex-dividend, leakage check 1, label balance, feature quality）

### Phase 2 完成評估準則
- [ ] 18 模型中至少 12 個 AUC > 0.52（quality gate）
- [ ] Ensemble 模型 AUC 優於所有基礎模型平均值 ≥ 2%
- [ ] Top-3 per horizon 的 OOD 測試 AUC 不下降 > 15%
- [ ] 無模型在 5 個壓力情景全部失敗
- [ ] Feature importance 分佈合理（無單一特徵占 > 50%）

### Phase 3 完成評估準則
- [ ] DSR 日報自動產出且格式一致
- [ ] Circuit breaker 能在 2 小時內觸發（有異常時）
- [ ] UI dashboard 無崩潰，load time < 3 秒
- [ ] 模型卡片涵蓋所有必要欄位
- [ ] Demo：給 5 個測試使用者，80% 表示可理解建議邏輯

## 十、後續可選優化方向（Phase 3+ 願景）

1. **序列模型**: LSTM / Transformer 捕捉時序依賴
2. **多市場**: 擴展至美股、加密貨幣
3. **即時預測**: WebSocket 推送而非日報
4. **A/B 測試**: 比較不同 DSR 表述對使用者決策的影響
5. **強化學習**: 動態調整部位大小與持股期限
6. **因子庫**: 套用經典量化因子（Fama-French, Momentum, Quality）
7. **多資產**: 期貨、選擇權希臘值整合
