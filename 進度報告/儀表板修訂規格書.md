# Part I. 專案定位、階段邊界與判讀原則

## 1.1 本文件的目的

本文件的目的不是只提供 UI 美化建議，而是要將目前專案的真實定位、已完成能力、未完成能力、雙面板設計邏輯，以及網站應如何與研究內容保持一致，整理成一份可直接執行的規格書。

本文件服務的對象有兩個：

1. **Agent / 開發端**：需要知道哪些內容可以改、哪些話可以說、哪些不能過度包裝。  
2. **專案團隊 / 規劃端**：需要明確掌握目前網站與專案應該如何對外呈現，避免「研究現況」、「網站文案」與「未來藍圖」混在一起。

## 1.2 專案真實定位（必須統一）

本專案的正確定位應統一為：

> **固定歷史資料集下的台灣股市多因子預測導向決策輔助系統課程專案**

這句話的意思很具體：

- 這是一份 **course project**，不是商業化投顧產品。  
- 這是一套以 **固定歷史資料期間** 為基礎的研究與展示系統，不是即時資料平台。  
- 這是一個 **決策輔助系統（decision-support system）**，不是自動下單系統。  
- 這個專案的核心價值是：**資料處理嚴謹性、方法論完整性、模型驗證流程、風險揭露能力，以及研究結果的可視化轉譯能力。**

## 1.3 目前完成到哪裡：Phase 1 / 2 / 3 的界線

### Phase 1：已完成的核心範圍

目前可確認已完成的內容包括：

- OHLCV 補齊（若檔案存在）  
- 股價前處理（缺值、停牌、除權息驗證、漲跌停偵測）

- 五支柱特徵工程  
- 洩漏偵測  
- Feature Store 與品質報告輸出

### Phase 2：已完成的核心範圍

目前可確認已完成的內容包括：

- Walk-forward 切分

### Phase 3：目前尚未完全完成，必須獨立判讀

根據先前報告與規劃脈絡，Phase 3 涵蓋的方向偏向：

- 模型治理（governance）  
- 風控與訊號治理  
- 更完整的 Model Card / Responsible AI 呈現

因此，**凡是屬於治理、風控、動態監控、進階解釋層、產品化處理的缺口，原則上先視為「Phase 3 尚未完成」的合理缺口。**

### Phase 判讀原則（Agent 必須遵守）

後續任何修改都必須先套用這個判讀順序：

1. 先確認該功能是否已在 Phase 1 / 2 真實實作。  
2. 若沒有，判斷它是否其實屬於 Phase 3 規劃能力。  
3. 若屬於 Phase 3，前台文案與 README 只能寫成：
   - 規劃中  
   - 未來擴充方向  
   - 設計理念 / 後續目標  
4. 不得把 Phase 3 願景內容包裝成「目前已經完成的功能」。

## 1.4 專案最重要的總原則

目前整個專案最值得保留、也最應被提升為網站與文件共同原則的一句話是：

> **寧可不知道，也不要誤導。**

這句話應成為所有前台頁面的總控制原則。具體來說：

- 若模型信心不足，應顯示「中性 / 觀望」，而不是勉強推薦。  
- 若資料不足，應顯示「資料不足，暫不做明確解讀」。  
- 若當前能力只支持歷史快照回看，就不能寫成即時推薦。  
- 若某項功能尚屬 Phase 3 規劃，就不能包裝成現有能力。  
- 若某個績效只在特定成本情境下成立，不能省略條件直接宣傳。

## 1.5 專案不是什麼（對外敘事紅線）

目前網站與 README 不應暗示本專案是：

- 即時投資建議服務  
- 即時市場監控平台  
- 可以保證報酬的推薦系統  
- 精準到盤中買賣點的交易系統  
- 持續自動更新的 live signal engine（若 pipeline 尚未完成）

## 1.6 專案是什麼（對外正確敘事）

目前網站與 README 應一致強調：

- 這是一個 **以固定歷史資料為基礎的互動式分析與研究展示平台**。  
- 這是一個 **以多因子預測與時序驗證為核心的決策輔助型課程專案**。  
- 這是一個 **同一套研究引擎，透過兩種資訊翻譯層對應不同使用者** 的系統。  
- 這個專案展示的不只是模型結果，而是完整的研究流程與嚴謹性。

# Part II. 專案內容與資料處理方式：嚴謹確認後的系統骨架

## 2.1 資料來源與資料期間

目前可確認的核心資料來源包括：

- `companies`：公司基本資料  
- `stock_prices`：每日股價資料  
- `income_stmt`：損益表資料  
- `stock_text`：新聞 / 公告文本  
- 補充 OHLCV：由 FinMind API 補齊

資料期間目前以 **2023-03-01 至 2025-03-31** 為主要分析範圍。

### 這個事實代表的網站敘事限制

由於目前資料期間為固定歷史區間，因此網站最多只能誠實地說：

> 本平台是一套基於固定歷史資料期間的互動式分析平台，可用於回看不同日期與不同 horizon 下的模型判讀與策略表現。

不能說：

- 目前推薦是即時市場訊號  
- 目前顯示的是今天市場最新狀態  
- 這是持續更新中的 live dashboard  
- 這是每週自動更新的最新推薦（除非後端 pipeline 真的完成）

## 2.2 Phase 1 的真實資料處理流程（應作為方法論核心亮點）

目前可確認的 Phase 1 流程可整理為：

1. 載入資料與設定  
2. 進行資料品質概覽  
3. 處理股價資料  
   - 缺值處理  
   - 停牌日過濾  
   - 除權息驗證  
   - 漲跌停偵測  
   - 股價品質檢查  
4. 處理文本資料  
   - 類型處理  
   - 過濾 / 去重 / 清洗  
5. 處理財報資料  
   - 累計轉單季  
   - PIT 日期對齊  
   - 財務比率計算  
6. 生成標籤  
   - D+1 / D+5 / D+20  
7. 進行五支柱特徵工程  
8. 進行前瞻偏差 / 洩漏偵測  
9. 輸出 Feature Store 與品質報告

### 對網站來說，這一段最該被突出的不是程式細節，而是三件事：

1. **資料不是直接丟進模型，而是有嚴格前處理。**  
2. **財報有做 PIT 對齊，避免偷看未來。**  
3. **特徵生成前後有做品質與洩漏檢查。**

## 2.3 PIT 與洩漏控制是目前最重要的方法論亮點之一

從現有程式與審核報告來看，專案在「避免資訊洩漏」上是有明確設計與實作痕跡的。這是整個專案在學術 / 評審視角下非常重要的加分點，必須在專業面板與首頁方法論說明中被強調。

網站與文件應清楚傳達：

- 財報特徵不是隨便對齊，而是根據公開時點做 PIT 處理。  
- 特徵不是用全資料回頭計算，而是有時序邏輯。  
- 訓練 / 驗證不是隨機切分，而是走時序式 walk-forward。

這部分是本專案最能顯示「不是只是做一個漂亮前台」的地方。

## 2.4 文本與事件資料：可以主張有清洗與事件訊號，但不宜誇大為成熟 NLP 引擎

從現況來看，可以合理主張：

- 專案納入了文本資料  
- 文本資料經過清洗、去重與事件性處理  
- 文本支柱作為事件 / 敘事訊號的一部分，已被納入整體特徵工程

但目前網站不宜把文本部分包裝成：

- 高成熟度新聞情緒 AI 引擎  
- 已完成全面 NLP 語意理解系統  
- 可精準解讀每一則新聞情緒方向的模型

若要寫進網站，建議用語是：

- 已納入新聞 / 公告類文本事件訊號  
- 現階段以文本清洗、事件密度與基礎事件特徵為主  
- 更進一步的文本語意分析可列為 Phase 3 / 後續擴充方向

## 2.5 財報與估值訊號：有價值，但要誠實面對覆蓋率限制

交叉審核報告指出：

- EPS YoY 覆蓋率偏低  
- 基本面特徵有相當比例的自然缺失值  
- 估值特徵也存在一定比例的 NaN

這些不一定代表資料錯誤，因為樹模型可處理 NaN，但代表網站必須更誠實：

- 新手頁不能把「基本面支持」說得像百分之百完備。  
- 專業頁應有資料覆蓋限制提示。  
- 若單一標的某些基本面資料不足，系統應顯示「資訊不足」而不是硬下結論。

## 2.6 Phase 2 的真實研究骨架

目前 Phase 2 可確認的真實方法論重點包括：

- Walk-forward 時序切分  
- 使用第一個 fold 的訓練集做 feature selection  
- D+1 / D+5 / D+20 三個 horizon 個別建模  
- LightGBM / XGBoost 雙引擎  
- ensemble  
- IC / ICIR  
- 三種成本情境回測  
- Quintile / Bootstrap / Drawdown / Alpha Decay  
- 排列檢定 / DSR / OOD 壓力測試  
- Platt Scaling 機率校準

這代表：

本專案在目前已完成的範圍內，其實並不是「只有訓練一個分類器然後把結果丟到前台」，而是一套相對完整的研究與驗證流程。

### 對網站而言，Phase 2 最該被講清楚的是四件事：

1. 不是 random split，而是時序式驗證。  
2. 不是只看 AUC，還有 ICIR、回測、成本、穩定性與統計檢驗。

# Part III. 網站架構與雙面板：重新整合後的完整建議

## 3.1 雙面板必須保留，而且應升級為整個專案的核心設計

你當初做雙面板的出發點是正確的，而且現在應該被正式寫成整個專案的核心架構。

因為這個專案真正厲害的地方，不只是模型，而是：

> **同一套研究引擎，能夠被翻譯成兩種不同層級的決策語言。**

專業使用者需要：

- AUC / LogLoss / Balanced Accuracy  
- Fold 穩定度  
- IC / ICIR  
- Quintile monotonicity  
- Bootstrap CI  
- Drawdown / Alpha decay  
- Calibration / OOD / DSR  
- 研究邏輯與品質門檻

新手使用者需要：

- 這檔股票大致偏多還是偏保守  
- 哪個 horizon 比較符合他的交易節奏  
- 交易成本會不會侵蝕報酬  
- 公司在做什麼  
- 有沒有基本面或估值支撐  
- 風險高不高  
- 現在比較像可研究、觀望，還是風險偏高

這兩者不是程度差異，而是語言差異。

## 3.2 雙面板的正式定義

### 面板 A：投資解讀面板（Decision Interpretation Panel）

定位：

- 面向較廣泛使用者  
- 將模型結果翻譯成使用者可理解的投資解讀資訊  
- 提供歷史快照下的判讀示範  
- 重點不是展示研究細節，而是協助理解「這檔股票大概該怎麼看」

### 面板 B：量化研究工作台（Quant Research Workbench）

定位：

- 面向教授、助教、評審、分析師、對方法論有興趣者  
- 完整呈現研究流程、方法論、指標、圖表與限制  
- 重點不是簡化，而是保留完整的研究證據層

## 3.3 首頁分流的正確敘事

首頁不應只是兩個跳轉按鈕，而應清楚告訴使用者：

- 兩個面板不是兩套不同系統  
- 使用的是同一套資料基礎、同一套模型與同一份研究引擎  
- 差別只在於資訊翻譯深度與呈現方式不同

### 首頁應明確寫出的核心句型

建議首頁加入類似下列敘事：

> 本平台基於同一套台股多因子研究引擎，提供兩種閱讀模式：一種是協助理解投資意涵的「投資解讀面板」，另一種是完整呈現模型指標、回測與研究證據的「量化研究工作台」。

## 3.4 新手面板應改名、改定位、改語氣

### 新手面板不應再被定義為「推薦頁」

新手頁若過度使用「推薦」、「前五名精選」、「買得安心」等語氣，會和目前固定資料集、歷史回測、學期專案定位產生衝突。

因此，新手頁應重新定義為：

- 投資解讀面板  
- 歷史快照解讀頁  
- 模型判讀示範頁

其本質是：

> **把研究結果翻譯成新手能理解的決策資訊。**

### 新手面板可以回答的問題

目前系統在 Phase 1 / 2 能力下，新手面板可以合理回答：

- 這個 horizon 下的方向大概偏什麼  
- 信心高低  
- 這家公司大致做什麼  
- 有沒有基本面 / 估值 / 事件上的支撐  
- 成本對這種操作週期的影響  
- 風險提醒與觀望建議

### 新手面板不能誠實承諾的事

目前不應承諾：

- 即時買點與賣點  
- 盤中操作建議  
- 保證報酬率  
- 真正 live 的動態更新推薦  
- 完整配息投資判斷（若資料未完整建模）

## 3.5 新手面板的資訊架構（強制規格）

每一張股票卡片或股票詳情頁，應依以下層級呈現：

### Layer 1：方向與信心（預設顯示）

- Horizon：D+1 / D+5 / D+20  
- 方向：偏多 / 中性 / 觀望 / 偏保守  
- 信心：高 / 中 / 低

### Layer 2：公司輪廓

- 公司名稱 / 股票代號  
- 所屬產業  
- 公司簡介  
- 市場別（若有）

### Layer 3：模型為何偏這樣判讀

以自然語言簡述：

- 近期價格動能偏強 / 偏弱  
- 基本面資料顯示成長 / 獲利表現（若資料足夠）  
- 估值相對偏高 / 偏低（若資料足夠）  
- 近期事件 / 新聞關注度偏高（若有）  
- 若資料不足，明確顯示：資料不足，暫不做明確解讀

### Layer 4：成本與風險

- 交易成本影響（高 / 中 / 低）  
- 波動風險  
- 流動性風險  
- 若高風險或可執行性不足，需加醒目標籤

### Layer 5：歷史觀察值（僅 historical mode 可顯示）

若顯示 forward return 或事後表現，必須明確標示：

- 這是歷史觀察值  
- 非當時可見資訊  
- 不代表即時預測承諾

## 3.6 新手面板的 UX 原則：漸進式揭露

新手頁不應一次塞滿所有圖表與數字，而應採用 progressive disclosure：

- 第一層先給方向、信心、風險  
- 第二層再給公司背景與簡化解釋  
- 第三層才展開更多技術細節  
- 進階資訊放在 expandable sections / tabs 中

這樣做的目的，是讓「看不懂量化指標的人」也能理解重點，同時保留進一步探索的空間。

## 3.7 專業面板的正確角色：證據層，而不是裝飾層

專業面板的存在不是因為「專業的人比較厲害」，而是因為這裡承擔了整個專案最重要的任務：

- 展示研究證據  
- 展示方法論  
- 展示風險與限制  
- 告訴評審這不是黑箱推薦頁

因此，專業面板必須保留並強化以下內容：

- Model Metrics  
- IC / ICIR  
- Backtest  
- Feature Analysis  
- Data Explorer  
- Fold stability / fold timeline  
- Calibration  
- Quality Gates / warnings  
- 限制條件與假設

## 3.8 專業面板後續最重要的優化方向

專業面板目前最重要的優化，不是加更多圖，而是：

1. **source of truth 一致化**  
   - 首頁數字、專業頁數字、README、JSON 報告要同步。  
2. **現況與規劃分層顯示**  
   - 已完成 / 規劃中 需要分開。  
3. **模擬數據或示意圖明確標示**  
   - 不能讓人誤以為所有數據都來自正式回測輸出。  
4. **限制條件置頂**  
   - 成本敏感性、資料期間、固定歷史資料、非即時等應明顯可見。

## 3.9 固定資料集下的「動態分析」應重新定義為歷史互動分析

目前不應把網站說成「動態即時分析平台」，但完全可以把它定義為：

> **Historical Interactive Analysis Dashboard**

也就是：

- 可切換 snapshot date  
- 可切換 horizon  
- 可切換產業 / 公司 / 市場  
- 可切換成本情境  
- 可回看不同時點下模型判讀與表現

### 對外正確說法

建議首頁與 README 使用這種語氣：

> 本平台為固定歷史資料集下的互動式分析與研究展示系統，可依不同日期與預測週期回看模型判讀與策略表現，並非即時市場監控系統。

這樣：

- 保留互動性  
- 不誤導  
- 與資料現況一致  
- 符合課程專案邏輯

# Part IV. Agent 直接執行清單：必改、建議改、Phase 3 再做

## 4.1 Agent 執行總原則

Agent 在修改任何檔案前，必須先做以下判讀：

1. 這個內容是否已被 Phase 1 / 2 真實實作？  
2. 若未實作，是否其實屬於 Phase 3 規劃？  
3. 若屬於 Phase 3，必須改成：
   - future work  
   - 規劃中  
   - 後續擴充  
4. 不得將規劃能力包裝成現況能力。

## 4.2 P0：必須立刻處理的項目

### P0-1. 首頁與 README 定位重寫

把所有對外敘事統一成：

- 台股多因子預測導向決策輔助系統  
- 固定歷史資料集下的互動式研究展示平台  
- 課程專案成果  
- 非即時、非投資建議、非保證報酬

### P0-2. 新手頁語義調整

把新手頁從「推薦頁」改寫為：

- 投資解讀面板  
- 歷史快照判讀頁  
- 模型解讀示範頁

必須移除或弱化任何太像即時投顧承諾的語句。

### P0-3. Historical / Prediction 語義切開

凡是使用 `fwd_ret_*`、歷史回看結果或事後表現的地方，必須加上明確標示：

- Historical observation  
- 非當時可見資訊  
- 不構成即時預測承諾

### P0-4. 加入資料期間與 snapshot 標示

首頁與兩個面板都必須清楚顯示：

- Data coverage period  
- Snapshot date / report date  
- 非即時資料提醒

### P0-5. phase 邊界說明

在網站或 README 某處加入：

- Phase 1：資料處理與特徵工程（已完成）  
- Phase 2：模型訓練與回測（已完成）  
- Phase 3：治理、風控、進一步監控與產品化強化（規劃 / 進行中）

避免使用者誤把未上線的治理功能視為系統缺陷。

### P0-6. 安全性處理：硬編碼 token

若 repo 內仍存在硬編碼 token：

- 立刻撤換 token  
- 改為 `.env` 載入  
- 清理敏感資訊暴露風險

## 4.3 P1：強烈建議調整的項目

### P1-1. 首頁統計與 KPI 改成動態同步

不要硬編碼：

- 股票數  
- 特徵數  
- horizon 數  
- quality gate 狀態  
- 最佳策略 / 最佳模型 / 最新報告日期

應從最新 JSON / metadata / report output 讀取。

### P1-2. README、報告、Dashboard 數字一致化

建立 single source of truth：

- 首頁展示數字  
- README 核心結果  
- 專業頁 KPI  
- 報告 summary

全部應由同一份最新產物驅動。

### P1-3. 新手頁加入「為何這樣判讀」的自然語言解釋

不能只列出股票與排序，應加入：

- 價格動能  
- 基本面 / 估值  
- 事件關注  
- 風險提醒

若資料不足，明確寫「資料不足」。

### P1-4. 成本敏感性顯示

新手頁應至少有簡化版說明：

- 短期策略成本侵蝕通常較高  
- 不同 horizon 的成本影響不同  
- 歷史表現需搭配成本情境解讀

### P1-5. 專業頁加入更多限制與警示標籤

例如：

- Fixed historical dataset  
- Not live market data  
- Benchmark simplification  
- Ensemble = simple average  
- Some governance capabilities remain in Phase 3

### P1-6. 新手頁強化例外處理

智慧選股看板 / 投資解讀面板最怕崩潰，因此：

- 加上頂層 try/except  
- 缺資料時顯示 graceful fallback  
- 不可因單一欄位缺失導致整頁失敗

## 4.4 P2：適合列為 Phase 3 的後續項目

以下內容若尚未穩定實作，應列入 Phase 3，而不是硬塞進現況頁面：

- 更完整的 Model Card  
- Regime 偵測整合到前台  
- 靜默模式 / fail-safe 前台呈現  
- 更成熟的可執行性檢查  
- 更完整的風控層與治理層  
- 更即時化的更新流程（若未來資料來源允許）  
- 更成熟的文本語意分析與敘事模組

## 4.5 實作順序建議（Agent 依序執行）

### Step 1：先改定位與文案

先統一：

- 首頁文案  
- README 簡介  
- 面板命名  
- 非即時 / fixed historical dataset 提示

### Step 2：再改新手面板結構

把新手頁改成「投資解讀面板」：

- 方向 / 信心  
- 公司輪廓  
- 為何這樣判讀  
- 成本與風險  
- 歷史觀察值（若顯示，需加標註）

### Step 3：整理專業面板的 source of truth

- 統一數字來源  
- 標示示意圖 / 模擬圖  
- 強化限制條件

### Step 4：最後再處理 Phase 3 標示與 roadmap

- 加入「已完成 / 進行中 / 規劃中」區塊  
- 讓使用者知道哪些不是缺陷，而是後續階段尚未完成

## 4.6 最終交付目標

Agent 完成調整後，網站與專案對外應達到以下狀態：

1. **定位一致**  
   - 網站、README、報告、頁面文案不再互相打架。  
2. **雙面板合理**  
   - 看得出來是同一研究引擎的兩種資訊翻譯層。  
3. **不誤導**  
   - 新手頁不再像即時投顧推薦頁。  
4. **研究價值更清楚**  
   - 專業頁成為證據層，而不是雜亂堆圖。  
5. **Phase 邊界清楚**  
   - 已完成的是 Phase 1 / 2，缺口若屬 Phase 3，會被正確標示。  
6. **更符合課程專案評分邏輯**  
   - 強調方法論嚴謹、資料處理、驗證流程與結果轉譯能力。
