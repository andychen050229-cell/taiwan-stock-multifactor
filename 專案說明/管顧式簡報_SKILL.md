# 管顧式簡報製作 · SKILL

> 本文件為 BCG / McKinsey / Bain 風格簡報的**嚴格製作規範**。每一條規則皆為強制紀律、不是建議。
> 適用情境：12 分鐘內結束的學術報告、商業提案、研究發表、董事會 deck。
> 目標：每張簡報只說一件事、每件事都有數字支撐、每個結論都可獨立稽核。

---

## 0 · 觸發時機（What this SKILL is for）

當下列任一條件成立時、強制套用此 SKILL：

- 製作 12 分鐘以內的簡報、deck、pitch、研究發表
- 撰寫含有量化結論、實驗結果、回測績效的 PPT / Keynote
- 為錄影、會議、董事會、學術報告等高密度場景生成材料
- 任何包含「研究 → 結論 → 部署」三段式論證的視覺化交付物

**不適用情境**：教學講義、自由創作、純圖文敘事、行銷海報。

## 1 · 標題層級規範（Title Hierarchy）

### 1.1 主標題（H1）—— **論述句、不是名詞片**

主標題是該頁的**核心命題**，必須是**完整的論述句**，回答「這頁要說什麼」。

**強制句型**（依優先順序）：

1. **「X 是 Y 的 Z」** — 定位句
   - ✓ 「D+20 是台股多因子預測從研究走向部署的可行切入點」
   - ✓ 「PSI 監控將模型衰退從事後檢討轉為事前預警」

2. **「X 並非 Y、而是 Z」** — 翻轉句
   - ✓ 「台股 alpha 並非不存在，而是被錯誤時序與單一因子框架遮蔽」
   - ✓ 「真正驅動模型的不是特徵數量，而是 Risk × Trend × Text 的訊號協同」
   - ✓ 「部署不是上線模型，而是建立可監控、可重訓、可停損的研究系統」

3. **「X 證明 / 顯示 Y」** — 證據句
   - ✓ 「四項獨立指標收斂，證明 D+20 訊號具樣本外穩定性」
   - ✓ 「LOPO 驗證顯示 Risk 支柱是模型不可替代的核心訊號源」

4. **「X 必須先 Y、才能 Z」** — 先決條件句
   - ✓ 「模型能力必須先通過時序紀律檢驗，才有資格被解讀為 alpha」

5. **「X、Y、Z」三段式格言**（用於收束頁）
   - ✓ 「部署條件具備、1–2 個月重新訓練、三大限制邊界」
   - ✓ 「DSR 12.12 ≫ 3.0 顯著、9 / 9 治理關卡 PASS」

### 1.2 主標題禁止句型

| 反例 | 為什麼錯 | 改寫 |
|---|---|---|
| 「方法論概覽」 | 純名詞片、無命題 | 「六階段 pipeline 將量化研究拆解為可追溯的交付流程」 |
| 「資料來源」 | 純名詞片 | 「七張資料表共同建立台股多因子模型的底層資料資產」 |
| 「結論」 | 無資訊量 | 直接寫結論本身 |
| 「研究背景」 | 起手式描述、不是命題 | 「台股短線 1–5 pp edge 在 Walk-Forward 下無法穩定複現」 |

### 1.3 副標題（H2 sub）—— **支援主標題的 SO WHAT 或 HOW**

副標題是主標題的**證據壓縮**或**機制說明**，必須與主標題形成上下文：

**強制要求**：
- 一句話、句末加句號（。）
- 字數 15 – 25 字（上限 30 字）
- 提供主標題的 HOW 或 SO WHAT，不重複主標題的 WHAT
- 用「。」終止；不用「！」或省略號

**範例對照**：

| 主標題（WHAT） | 副標題（HOW / SO WHAT） |
|---|---|
| 「四項獨立指標收斂，證明 D+20 訊號具樣本外穩定性」 | 「AUC、ICIR、DSR、Top 1% hit rate 同步成立。」 |
| 「Top 1% 高信心訊號能把預測優勢轉化為策略績效」 | 「回測顯示模型具經濟性，但需維持一致成本口徑。」 |
| 「PSI 監控將模型衰退從事後檢討轉為事前預警」 | 「Drift governance 是部署後的必要條件。」 |

### 1.4 副標題禁止內容

- ❌ 純粹的 metric inventory：「預測強度（AUC）、IC 穩定性（ICIR）、多重檢定（DSR）、選股效益（Top 1%）」
- ❌ 第一人稱起手：「我們的切點是⋯⋯」
- ❌ 描述方法 setup：「把模型機率分數化為實單部位、月再平衡⋯⋯」（這是 method 描述、不是 takeaway）
- ❌ 多句連發：副標題只能一句

## 2 · 字體與字級紀律（Typography Discipline）

### 2.1 字體統一規範

**強制：所有文字使用「微軟正黑體（Microsoft JhengHei）」**

```python
# python-pptx 設定
LATIN_FONT = 'Microsoft JhengHei'   # 拉丁字母
EA_FONT    = 'Microsoft JhengHei'   # CJK 字元
MONO_FONT  = 'Microsoft JhengHei'   # 數字 / 縮寫
```

**為什麼**：
- 微軟正黑體在 Windows / macOS / Office / 投影設備全平台一致
- CJK + Latin 混排的字寬平衡優於 Inter / Helvetica
- 數字節奏自然、不需要切換 Mono 字體
- 教學 / 學術場合最不出錯

### 2.2 最小字級紀律

**強制：所有可見文字 ≥ 12pt**

| 元素 | 最小字級 | 建議字級 |
|---|---|---|
| 主標題（H1） | 36pt | **40pt 統一** |
| 副標題（H2 sub） | 18pt | **18pt 統一** |
| 章節 eyebrow | 12pt | 14pt |
| 卡片標題 | 16pt | 18 – 22pt |
| 卡片副標 | 12pt | 13 – 15pt |
| 正文 / 圖表標籤 | **12pt（紅線）** | 13 – 15pt |
| Footnote / Source 行 | 12pt | 12pt |
| 大號 metric（hero number） | 28pt | 30 – 80pt |

**禁止**：
- ❌ 任何文字小於 12pt（連 axis labels、bar chart 數字、source 引用都不行）
- ❌ 為了「塞進框內」把字級從 13pt 降到 11pt——應該是改縮寫、改換行、加大框、減少內容
- ❌ 用斜體（italic）作為視覺強調——改用粗體、改色、改大小

### 2.3 字級階層對應關係

```
40pt  ─── 主標題
26pt  ─── 標題彩色延續（如 "5-Layer Stack · Storage → ETL → ..."）
22pt  ─── 卡片大標
18pt  ─── 副標題、卡片中等大字
16pt  ─── takeaway bar 主句
14pt  ─── eyebrow chip / 章節標籤
13pt  ─── 卡片內文
12pt  ─── footer / source / footnote (絕對下限)
```

每張頁面同類元素必須使用同一字級。**跨頁面類型一致 > 單頁美感**。

### 2.4 強調手段

僅以下方式可用作強調：

1. **加粗**（bold）—— 重要詞彙、metric 數字
2. **改色**—— accent color（NAVY / TEAL / EMERALD / ROSE / GOLD）僅限數字、結論詞、品牌
3. **改大字級**—— hero metric

**禁止強調手段**：
- ❌ 斜體（italic）
- ❌ 底線（underline）
- ❌ 全大寫中文
- ❌ 反白色塊
- ❌ 加陰影 / 發光（呼吸燈例外、僅限主導指標）

## 3 · 結構與節奏（Structure & Pacing）

### 3.1 三幕劇結構（Three-Act Discipline）

主簡報必須分為三段、每段 3 – 4 頁、共 10 – 11 主頁：

| 幕 | 名稱 | 頁面職責 | 典型頁數 |
|---|---|---|---|
| Act 1 | **SETUP** | 切角、方法、研究立場 | 3 頁（P.02 – P.04）|
| Act 2 | **FINDINGS** | 證據、攻擊、發現 | 3 – 4 頁（P.04 – P.07）|
| Act 3 | **ACTION** | 部署、邊界、收束 | 3 – 4 頁（P.08 – P.11）|

**頁面下角必須有三幕指示燈**：
- 三個圓點橫排、當前幕亮色
- 旁邊小字標籤 SETUP / FINDINGS / ACTION

### 3.2 Pyramid Principle 應用

每頁的視覺由上而下對應 Minto Pyramid：

```
┌─────────────────────────────────────────────────┐
│  EYEBROW  · 章節標籤、定位（14pt mono）           │
│  H1 主標題（40pt）—— 命題層 WHAT                │
│  H2 副標題（18pt）—— 機制 / 結論層 HOW         │
│  ─── Rule divider ───                            │
│  視覺主體（圖、表、卡片網格）—— 證據層 EVIDENCE │
│  Takeaway bar（16pt）—— 收束 SO WHAT          │
│  Source · 自行整理 · paper-ref（12pt 灰）         │
└─────────────────────────────────────────────────┘
```

每頁讀者在 5 秒內應掌握 H1 + 副標 + Takeaway bar 的三層結論；所有圖表只是支撐。

### 3.3 收束 takeaway bar

每頁底部必須有色塊收束帶（takeaway_bar）、顯示該頁的 SO WHAT：

- 標籤：mono 大寫小字（11pt）BOTTOM LINE / WHY THIS MATTERS / SEQUENCING / KEY POINT 等
- 主句 16pt、可分 2 – 3 行
- 結尾可帶 → Appendix 指引、用 INK_3_DK 灰色 14pt
- 高度公式：`label_height(48 if 有 label else 18) + 行數 × 28 + 8 buffer`

### 3.4 配速 budget

12 分鐘錄影、又萍語速 **280 字 / 分**：

| 頁面類型 | 時長 | 字數（CJK + ENG×2 + DIG）|
|---|---|---|
| 封面 | 0:50 | ~250 |
| 主論述頁（標準）| 1:05 – 1:15 | ~290 – 360 |
| 收束 / 結尾頁 | 0:25 – 0:35 | ~120 – 160 |

**強制：每頁逐字稿開頭用「本頁的命題是」/「結論先給」/「來看 X」一句 anchor**，3 – 5 秒內把命題層浮出。

## 4 · 內容紀律（Voice Rules）

### 4.1 BCG / McKinsey 口吻五原則

1. **Metric-first**：每個論述後緊跟數字
   - ✓ 「Risk 是地基——拿掉 Risk，AUC 掉 138.6 bps」
   - ✗ 「Risk 是地基」

2. **「並非 / 印證 / 則 / 一句話」式轉折**
   - 「但這並非代表 alpha 不存在」
   - 「印證」、「換言之」、「下一步是」、「一句話」

3. **、 chains for parallel facts**
   - 「結論並非運氣、並非過擬合、可獨立稽核」

4. **負面證據自信地講**
   - 不掩蓋限制：直說「不適用 intraday、訊號半衰期不足」

5. **學術 disclaimer 法律邊界明確**
   - 投資相關頁必出現「本研究為學術成果、不構成投資建議」
   - 不可隱含於正文、必須獨立成行

### 4.2 禁止用字（中國語感 / AI 感）

| 禁字 | 用詞特性 | 替換 |
|---|---|---|
| 萬靈丹 | sales pitch | 通用解 / 解決方案 |
| 開閥門 | 業務行銷腔 | 落地 / 開始試運行 |
| 全鏈路 / 全棧 | 中國行業 jargon | 完整流水 / 端到端 |
| 賦能 | 中國商業語感 | 支援 / 提供 |
| 拉滿 | 中國口語 | 達到上限 / 全速 |
| 做穩做深做久 | 直銷腔 | 持續可監控 |
| 顆粒度 | 中國技術 jargon | 解析度 / 細緻度 |
| 數據 | （簡體用法） | **資料**（繁體標準） |
| 信息 | （簡體用法） | **資訊** |
| 視頻 | （簡體用法） | **影片** |
| 高大上 | 中國口語 | 專業 / 嚴謹 |
| 一鍵 | 行銷腔 | 一次操作 / 自動 |
| 賦權 / 加持 | 中國商業 | 強化 / 配合 |

### 4.3 數字書寫紀律

| 場景 | 寫法 | 範例 |
|---|---|---|
| 百分點差距 | pp（不寫 ppt）| +10.53 pp |
| 百分比 | % | 36.81% |
| 基點 | bps | 138.6 bps |
| 千位分隔 | 半形逗號 | 948,976 |
| 小數點 | 半形句點 | 0.6455 |
| 中文千 / 萬 | 不混用 | 「948,976 樣本」非「94.9 萬樣本」 |
| 範圍 | 用全形破折號 | 0.633 – 0.656、1 – 2 個月 |
| 單位 | 半形空格隔開 | `12.12` 不寫 `12.12bps` |

### 4.4 縮寫展開規則

**第一次出現**：簡體中文括號或破折號補充（如目標讀者非量化背景）

```
DSR——也就是多重檢定後仍顯著的 Sharpe——遠超過 3.0 的顯著性門檻
```

**第二次以後**：直接用縮寫，假設讀者已理解

**通用縮寫不需展開**：AUC、ML、CV、API、SQL、JSON、PPT、IDE 等

## 5 · 中文換行紀律（CJK Line-Break Discipline）

### 5.1 基本原則

| 標點 | 處理 | 原因 |
|---|---|---|
| 句號（。）| **強制換行** | 完整論述結束、視覺呼吸 |
| 分號（；）| **建議換行** | 兩個獨立子句 |
| 逗號（，、、）| **不換行**、自然 wrap | 同一語意流動 |
| 問號（？）| 情境決定（句末通常換行）| |

### 5.2 反例與修正

| 反例 | 為什麼錯 | 修正 |
|---|---|---|
| `以 9 支柱因子棋盤、{br}948,976 樣本、{br}505 交易日、{br}D+20 預測` | 逗號強制換行、視覺斷碎 | 整句流動、自然 wrap |
| `本系統適用 D+20 中長線。不適用 intraday；訊號半衰期不足` | 句號 + 分號混用、節奏混亂 | 句號斷句、保持兩段平行 |
| `PASS。本研究為學術成果不構成任何投資建議` | 句號後未換行、disclaimer 與正文擠 | disclaimer 必獨立成行 |

### 5.3 標點全形 / 半形

- **CJK 句中**：全形（、，。：；！？）
- **CJK + 拉丁混用**：全形 CJK 標點 + 半形拉丁標點
- **純拉丁段**：半形（,.;:!?）
- ✗ 禁止：CJK 句中用半形逗號（如「我們、948K 樣本, 505 交易日」）

## 6 · 配色紀律（Color Discipline）

### 6.1 設計 Token

| Name | Hex | 用途 |
|---|---|---|
| NAVY | `#1E3A8A` | 主色、強調數字 |
| TEAL | `#06B6D4` | 副色、結論詞 |
| EMERALD | `#10B981` | 正向 / 通過 |
| ROSE | `#F43F5E` | 負向 / 失敗 / 警示 |
| GOLD | `#F59E0B` | 警告 / 條件通過 |
| INK | `#0F172A` | 主標題、強內文 |
| INK_2 | `#334155` | 次內文 |
| INK_3 | `#64748B` | 副註、source |
| INK_3_DK | `#475569` | 副註強調（取代斜體）|
| INK_4 | `#94A3B8` | 標籤、disabled |
| PAPER | `#FFFFFF` | 主背景 |
| TINT | `#F1F5F9` / `#ECFEFF` | 卡片底色 |

### 6.2 配色紀律

1. **每張投影片色相 ≤ 3 種主色**（NAVY 必出現、再加 1 – 2 個 accent）
2. **數字才上色**——一般文字用 INK / INK_2，accent 色保留給 metric / 結論詞
3. **負面結果用 ROSE，正面用 EMERALD**——不可互換
4. **不使用「鮮紅 + 鮮綠」對比**——色盲不友善
5. **avoid rainbow**——3 種以上 accent 同時出現視為錯誤

## 7 · 視覺元素規範（Visual Element Specs）

### 7.1 畫布規格

- **解析度**：1920 × 1080（16:9）
- **EMU 換算**：96 DPI、1 pixel = 9525 EMU
- **左右邊距**：96 px
- **上下邊距**：38 px（chrome 帶）、1018 px（source 行 y 座標）

### 7.2 Chrome 帶（頁眉）

```
[TW] 多因子股票預測系統  · v11.5.18              <Page Title>     [No 02]
                                                                   ↑ 移至底部右下
```

- 左：Logo chip（30×30 NAVY）+ 系統名 + 版本
- 右：頁面標題 mono 12pt（文字向右對齊）
- **頁碼放底部右下** `(x=1780, y=1018)` —— 統一不變
- **附錄 chrome**：Logo chip 改為 ROSE「APX」、頁碼改 ROSE 顏色

### 7.3 Eyebrow 章節標籤

- 位置：`(x=96, y=112)`、寬 700px
- 樣式：mono 14pt、bold、TEAL（主簡報）/ ROSE（附錄）
- 格式：`— PROBLEM · 為什麼鎖定 D+20`（前帶長破折號）

### 7.4 主標題位置

- y=142 起、容器寬 1728px、高 130px
- 字級 40pt、字色 NAVY，accent 部分 TEAL / EMERALD / ROSE
- 主標 + 副標總高度 ≤ 130px

### 7.5 Rule divider

- y=280（主簡報）/ y=286（附錄）
- 寬 80px、高 4px、色 TEAL（一律）

### 7.6 Source 行

- 位置：`(x=96, y=1018)`、寬 1300px
- 樣式：mono 12pt、INK_3 灰、ls=0.5 字距加大
- 格式：`Source · 自行整理 · López de Prado (2018) §7.4`
- 內部 artifact（phaseN_report、*.parquet）一律寫「**自行整理**」
- 外部論文保留全名 + 年份

### 7.7 三幕指示燈

- 位置：`(x=1500, y=1024)`、3 點橫排、間距 22px
- label：SETUP / FINDINGS / ACTION（向右對齊）
- 當前幕：實心圓 5px TEAL；其餘：空心圓 4px

### 7.8 Card 容器

- 圓角：0px（直角、嚴肅感）
- 邊框：1px RULE
- 背景：PAPER 或 TINT
- 頂部 stripe：4px 高彩色色帶（accent 色）、強調該卡屬於哪一支柱 / 結論層

## 8 · 圖表紀律（Chart Discipline）

### 8.1 圖表類型對應

| 用途 | 圖表類型 | 禁忌 |
|---|---|---|
| 跨類別比較 | 水平 / 垂直長條 | 圓餅圖（精度差）|
| 時序趨勢 | 折線圖 | 區域圖（除非堆疊有意義）|
| 分位 / 百分比 | 堆疊長條 / heatmap | 雷達圖 |
| 對應關係 | 散佈圖 | 3D 任何圖 |
| 比例 | Donut（中心 metric）| 圓餅圖 |

### 8.2 圖表內字級

- Axis tick labels：≥ 12pt
- Series labels：≥ 12pt
- Chart title：≥ 14pt
- 圖內 callout 數字：12 – 16pt

### 8.3 圖表配色

- 使用 design token 4 種以內
- 「best」/「主要」用 cyan #67E8F9 或 TEAL
- 對照 / baseline 用 INK_3 灰 + 虛線
- 警示用 ROSE、通過用 EMERALD

### 8.4 必須標註

- Random / baseline 線：dashed、ROSE、annotation
- Best item：★ 標記 + 顏色突出
- Y 軸範圍：包含 baseline，不允許「截尾誇大」

## 9 · 製作流程（Production Flow）

### 9.1 順序

```
1. 確定論述結構（10 – 11 主頁、3 幕劇）
2. 寫每頁的主標 + 副標（先寫文字、再做視覺）
3. 每頁設計：eyebrow → 主標 → 副標 → divider → 主體 → takeaway → source

5. 每頁 render PNG 1920×1080
6. 多視角審查（§ 11）
7. 修正 → re-build → re-render → 再審
8. 直到 95+ 分；最後 sync 至根目錄
```

## 10 · 多視角審查（Multi-Perspective Review）

每份 deck 完成後、強制執行三輪審查：

### 10.1 三位設計師視角

| 視角 | 檢查項 |
|---|---|
| **Layout designer** | 頁碼 / Source / chrome 跨頁一致；左右 margin 96px 對齊 |
| **Typography designer** | 字體 100% 微軟正黑體；無斜體；字級皆 ≥ 12pt；CJK 換行紀律 |

### 10.2 兩位 BCG / McKinsey 顧問視角

| 視角 | 檢查項 |
|---|---|
| **Pyramid principle** | 標題 = WHAT；副標 = HOW；takeaway = SO WHAT |
| **MECE / hypothesis-driven** | 三欄並列必 MECE；每頁回答一個假設；無 metric inventory 副標 |

### 10.3 兩位一般閱讀者視角

| 視角 | 檢查項 |
|---|---|
| **非量化背景** | 第一次出現的金融術語有 inline gloss；圖表標籤 ≥ 12pt |
| **嚴格讀者** | 每個 metric 都有來源；disclaimer 法律邊界明確；無 typo |

審查後產出「失誤清單」、依優先順序逐條修復。

## 12 · 配速與逐字稿（Pacing & Script）

### 12.1 逐字稿規範

- **配速**：280 字 / 分（Chinese characters per minute equivalent）
- **頁面開頭 anchor 句**：每頁第一句必為「本頁的命題是 X」/「結論先給」/「來看 X」（3 – 5 秒）
- **大數字逐字唸出**：「OOS AUC 零點六四五五」、「DSR 十二點一二」（不簡化）
- **disclaimer 必念**：「本研究為學術成果、不構成投資建議」（8 字落地）
- **Filler 禁止**：「這個」、「那個」、「就是說」、「對」、「啊」一律不用

### 12.2 inline gloss 原則

對非量化背景觀眾、第一次出現的金融術語必加簡釋：

```
ICIR——也就是預測強度的穩定性——數字越大代表預測越穩
```

- 用破折號或括號自然帶過、不像 disclaimer 那樣加重
- 同個術語只解釋一次（後續直接用）
- 課程已涵蓋（決策樹、SVM、AUC、CV）的 ML 術語**不需解釋**

### 12.3 計時表（必附）

逐字稿末尾必附：

| 段落 | 預估時長 | 累計 |
|---|---|---|
| P.01 Cover | 0:50 | 0:50 |
| ... | ... | ... |
| P.11 Closing | 0:30 | **11:25 – 12:00** |

附「若超時可省略 X / Y 段共 Z 秒」的 fallback 機制。
