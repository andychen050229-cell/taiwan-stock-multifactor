## 2. 設計哲學與 UX 原則

### 2.1 三條設計軸

```
    科技感                     資料密度                   易讀性
       |                           |                          |
       |  Bloomberg Terminal        |  Notion / Linear         |  Apple HIG
       |  glint.trade               |  Superhuman              |  Material 3
       |                           |                          |
       └─── 我們的定位點：亮色版的 glint.trade · 中等資料密度 · 高易讀性 ─┘
```

### 2.2 參考對標

| 來源 | 我們借用 | 我們捨棄 |
|---|---|---|
| **glint.trade / terminal** | 等寬數字、grid overlay、instant feel | 深色、資訊過密、只給專業交易員 |
| **Bloomberg Terminal** | 資訊層級、脈動 LIVE 點、單色 accent | 深色黑底黃字、學習曲線太陡 |
| **財報狗 / 玩股網** | 投資解讀面板的敘事性 | 資訊擁擠、過度廣告 |
| **Stripe Dashboard** | 亮色 + 紫藍漸層 + 大方留白 | 商務 SaaS 氣質（我們是金融） |
| **Apple Human Interface** | 層級節奏、字體可讀性、圓角一致 | 太克制、缺少科技感 |

### 2.3 五大設計價值主張

1. **Instant comprehension（立即理解）**：每個數字都附帶 context（前期比較 / 基線 / 閾值）
2. **Progressive disclosure（漸進揭露）**：首頁只給結論，深度細節放在子頁 Tab
3. **Trust through transparency（透明即信任）**：所有 KPI 都能追溯到 JSON 原始檔
4. **Dual-track experience（雙軌體驗）**：同一組資料，散戶看敘事、研究員看公式
5. **Tech as a signal of rigor（科技感作為嚴謹度的訊號）**：`JetBrains Mono` 數字 + grid background = 暗示「這是嚴格驗證的結果，不是隨便猜的」

### 3.2 字體系統

```css
--gl-font-sans: 'Inter', -apple-system, BlinkMacSystemFont, 'Segoe UI',
                'Microsoft JhengHei', 'PingFang TC', sans-serif;
--gl-font-mono: 'JetBrains Mono', 'SF Mono', 'Consolas', monospace;
```

**Type Scale**

| 層級 | size | weight | tracking | 用途 |
|---|---|---|---|---|
| H1（Hero Title） | 2.1 rem | 800 | −0.03 em | 頁首 Hero |
| H2（Section） | 1.5 rem | 700 | −0.01 em | 段落主題，左側漸層條 |
| H3（Subsection） | 1.15 rem | 600 | −0.01 em | 面板內標題 |
| Body | 1.0 rem (16 px) | 400 | 0 | 內文 |
| Small | 0.85 rem | 500 | 0 | 說明 / 副標 |
| Eyebrow | 0.76 rem | 600 | +0.06 em UPPERCASE | KPI 標籤、Tag |
| Numeric | inherit | 700 | tabular-nums | `stMetricValue` 強制 Mono |

**原則**：所有**數字** → Mono；所有**文字** → Inter + JhengHei fallback；tabular-nums 讓縱向數字對齊整齊。

### 3.3 間距與排版

基於 4-pt grid：`4 · 8 · 12 · 16 · 24 · 32 · 48 · 64 px`

| 用途 | 建議值 |
|---|---|
| 卡片 padding | `18px 20px` |
| 面板 padding | `20px 24px` |
| Hero padding | `40px 48px` |
| Gap（grid） | `20px` |
| H2 上下間距 | `2rem top · 1rem bottom` |
| 圓角 radius | 小元件 `10px` · 卡片 `14px` · 面板 `20px` |

### 3.4 陰影 / 層級（Elevation）

| Level | Token | 定義 | 用途 |
|---|---|---|---|
| 0 | — | 無陰影 | 區塊底色 |
| 1 | `--gl-shadow-sm` | `0 1px 2px rgba(15,23,42,.04), 0 1px 3px rgba(15,23,42,.06)` | 卡片預設 |
| 2 | `--gl-shadow-md` | `0 4px 6px .04, 0 10px 15px .06` | 面板 |
| 3 | `--gl-shadow-lg` | `0 10px 15px .06, 0 20px 25px .08` | Modal、Tooltip |
| 4 | `--gl-shadow-glow` | `0 0 0 1px rgba(37,99,235,.08), 0 8px 30px rgba(37,99,235,.12)` | Hover focus glow（品牌藍） |

### 3.5 動效原則

| 動效 | Duration | Easing | 觸發 |
|---|---|---|---|
| Card hover | 250 ms | `ease` | 滑鼠進入 |
| Button press | 150 ms | `ease-out` | 點擊 |
| Fade-in | 400 ms | `ease-in-out` | 頁面載入 |
| Pulse（LIVE 指示燈） | 2 s loop | `cubic-bezier(0.4,0,0.6,1)` | 恆動 |
| Gradient scan（頁首） | 6 s loop | linear | 恆動 |

**原則**：動效用於「強調狀態改變」與「表達即時性」，不用於裝飾。

## 4. 科技感設計手法

這一節詳述 6 個具體的「讓資料看起來像專業系統而非 Excel」的技巧。

### 4.1 Glass-morphism（玻璃擬態面板）

```css
.gl-panel {
    background: var(--gl-overlay);           /* rgba(255,255,255,0.72) */
    backdrop-filter: blur(12px);
    border: 1px solid var(--gl-border);
    border-radius: 20px;
    padding: 20px 24px;
    box-shadow: var(--gl-shadow-md);
}
```

**為什麼**：半透明 + 模糊讓面板「浮起」於 canvas 之上，視覺深度增加，同時保留背景網格可見，暗示「可疊加」的資訊層。

### 4.2 Tech-grid Background（科技網格背景）

```css
.block-container::before {
    content: "";
    position: fixed;
    inset: 0;
    background-image:
        linear-gradient(rgba(37,99,235,0.025) 1px, transparent 1px),
        linear-gradient(90deg, rgba(37,99,235,0.025) 1px, transparent 1px);
    background-size: 40px 40px;
    pointer-events: none;
    z-index: 0;
}
```

**為什麼**：40 px × 40 px 的藍色極淡方格線（opacity 0.025）讓背景有「工程圖紙」質感。不干擾閱讀，但任何滑動都讓使用者感受到「這是一個精密系統」。

### 4.3 Gradient Title（漸層標題）

```css
.gl-hero-title {
    font-size: 2.4rem;
    font-weight: 800;
    background: var(--gl-grad-pri);        /* blue → violet */
    -webkit-background-clip: text;
    background-clip: text;
    color: transparent;
}
```

**為什麼**：單色標題 = 普通文件；漸層標題 = 品牌認同。只在 Hero 使用，其他標題仍是純黑以保可讀性。

### 4.4 Pulse Live Indicator（脈動即時點）

```css
.gl-live {
    display: inline-flex; align-items: center; gap: 8px;
    font-family: var(--gl-font-mono);
    font-size: 0.76rem;
    color: var(--gl-emerald);
    text-transform: uppercase;
    letter-spacing: 0.1em;
}
.gl-live::before {
    content: "";
    width: 8px; height: 8px; border-radius: 50%;
    background: var(--gl-emerald);
    animation: gl-pulse 2s cubic-bezier(0.4,0,0.6,1) infinite;
}
@keyframes gl-pulse {
    0%, 100% { opacity: 1; box-shadow: 0 0 0 0 rgba(16,185,129,.7); }
    50%      { opacity: .8; box-shadow: 0 0 0 8px rgba(16,185,129,0); }
}
```

**為什麼**：一個小小的綠點在 Hero 上脈動 → 使用者直觀感受「這個系統是活的、資料是最新的」。心理效應遠強於寫一行「資料更新至 2026-04-20」。

### 4.5 Mono-font for Numbers（數字等寬）

所有數字用 `JetBrains Mono` + `tabular-nums`：
```
Sharpe 0.81         ← 若用比例字，小數點會跑位
        vs
Sharpe 0.81         ← Mono + tabular-nums，數字在縱向完美對齊
```

**為什麼**：量化研究員一眼看到 `0.015` 是 `+0.015` 還是 `-0.015` 非常關鍵。等寬數字配合 tabular-nums 能讓比較時眼睛不用跳來跳去。

### 4.6 Accent Bar on H2（章節左側漸層條）

```css
.block-container h2 {
    position: relative;
    padding-left: 16px;
}
.block-container h2::before {
    content: "";
    position: absolute;
    left: 0; top: 4px; bottom: 4px;
    width: 4px;
    background: var(--gl-grad-pri);
    border-radius: 4px;
}
```

**為什麼**：每個 H2 標題左側都有一條 4 px 寬的藍紫漸層條，視覺錨點明確，讓長頁面捲動時仍能快速定位段落。

### 8.1 設計動機

### 8.4 情緒分析的三層辯證

我們**不是**純粹靠 SnowNLP（它是在淘寶評論上訓練的，對財經術語弱），而是三層融合：

```python
# 偽代碼
base_score = snownlp.sentiments(text)          # [0, 1] 範圍

# 辭典加權
bullish_hits = count_terms(text, BULLISH_DICT)  # 創新高、大漲、獲利
bearish_hits = count_terms(text, BEARISH_DICT)  # 下跌、虧損、利空

adjusted = (
    0.5 * base_score
    + 0.3 * normalize(bullish_hits - bearish_hits)
    + 0.2 * (length_factor * source_weight)    # 權威媒體加權
)
```

### 8.5 文本特徵在模型中的邊際貢獻

根據 Phase 6 LOPO 驗證：

| 支柱 | Δ Edge (bps)│ 排名 | 說明 |
|---|---|---|---|
| risk | +1.39 | #1 | 風險因子最大邊際貢獻 |
| trend | +1.15 | #2 | 趨勢因子次之 |
| val | +0.84 | #3 | 估值因子 |
| event | +0.68 | #4 | 事件因子 |
| fund | +0.45 | #5 | 基本面 |
| chip | +0.33 | #6 | 籌碼 |
| ind | +0.26 | #7 | 產業 |
| **txt** | **+0.22** | **#8** | 文本（邊際小但穩定正向） |
| **sent** | **+0.14** | **#9** | 情緒（邊際最小） |

**如何解釋這個結果？**

1. **txt / sent 的邊際看似小**（+0.22 / +0.14 bps），但這是**在已有 89 個 price/fund 特徵的前提下**——本身能再擠出一點訊號就已經有價值
2. **文本邊際 > 情緒邊際**：說明「文章裡面提到什麼」（關鍵字）比「文章整體是正面還負面」（情緒）更有預測力
3. **符合金融文獻**：Loughran & McDonald (2011)、Tetlock (2007) 等學者都指出情緒比字典計數小
4. **未來改進空間**：擴充資料源（PTT / Dcard / 法說會逐字稿）可將邊際推高

## 9. 解釋性結果與呈現

### 9.1 四層解釋性

我們採用「Zoom out → Zoom in」的四層架構：

```
Level 1 支柱級（最抽象）     ←  LOPO 驗證
         ↓
Level 2 特徵級               ←  SHAP summary plot
         ↓
Level 3 樣本級               ←  SHAP force plot（單筆預測解釋）
         ↓
Level 4 個股級（最具體）      ←  2454 聯發科 OOS 深度案例
```

### 9.2 LOPO（Leave-One-Pillar-Out）支柱歸因

**方法**：9 支柱每次輪流拿走一個、重訓 6 個模型、重算 edge，比較「完整模型 − 缺支柱模型」就是該支柱的邊際貢獻（Δ Edge in bps）。

**為何不是純 SHAP？**
- SHAP 告訴你「這個特徵的 SHAP value 平均 0.x」——但不能告訴你「拿掉整個風險支柱會損失多少報酬」
- LOPO 直接衡量經濟價值（bps of edge），對評審 / 決策者更直觀

**在 Dashboard 上**：`🔭 Phase 6` 的 Tab 1：
- 水平長條圖，按 Δ Edge 由大到小排序
- 每個長條染其對應的支柱色
- Hover 顯示 `n_features / edge_full / edge_drop / delta_bps`

### 9.3 Threshold Sweep（出手率 ↔ 命中率）

**問題**：「我應該在模型預測機率 > 多少時買進？」

**方法**：在 [0.30, 0.50] 區間掃 21 個閾值，計算每個閾值的：
- `hit_rate`：命中機率
- `call_rate`：出手率（每天多少 % 的股票被選）
- `edge`：命中報酬 × 出手率 − 基線
- `n_picks`：樣本數

**在 Dashboard 上**：`🔭 Phase 6` 的 Tab 2 雙軸互動圖：
- 左軸（藍線）：hit_rate 隨 threshold 變化
- 右軸（橘線）：call_rate 隨 threshold 變化
- 底部填色：edge（最高點用標籤標註）
- 右下：Top-K precision 長條圖（K=5/10/20）

**解讀**：t=0.35 是最佳點——再高 hit 率更高但樣本太少不穩定；再低 call 率太高但 hit 率崩潰。

### 9.4 Single-Stock Deep Dive（2454 聯發科）

**為何選 2454？** 在 Phase 3 個案研究的四檔權值股中（2330/2317/2454/2303），聯發科的邊際效應最強。

**圖**：
- 上：累積命中 vs 日期 的階梯圖
- 下：月度命中率橫條圖 + 平均線

**關鍵數字**：
- OOS 213 日、累積命中 87 次、整體命中率 40.8%
- 月度平均命中率 61.5%、最佳月 73.3%
- Edge +17.13%（與整體策略 +17.77% 相符）

## 12. 部署架構

### 12.1 三層入口結構（Streamlit Cloud 向下相容）

```
┌──────────────────────────────────────────────────────────┐
│ Streamlit Cloud Main file path 可設任一：                 │
│                                                          │
│   Option A (當前部署):  dashboard/app.py                  │
│      → shim changes CWD → 程式碼/儀表板/app.py            │
│                                                          │
│   Option B (推薦新設):  程式碼/儀表板/app.py              │
│      → canonical, 無轉介                                  │
│                                                          │
│   Option C (fallback): streamlit_app.py                   │
│      → root-level shim → 程式碼/儀表板/app.py             │
└──────────────────────────────────────────────────────────┘
```

三個入口本質上都執行同一份 `程式碼/儀表板/app.py`，沒有 code duplication。

### 12.2 shim 原理

```python
# dashboard/app.py 與 streamlit_app.py 共用同一邏輯：
import os, sys
from pathlib import Path

DASHBOARD = Path(__file__).resolve().parent / "程式碼" / "儀表板"
APP_PATH  = DASHBOARD / "app.py"

sys.path.insert(0, str(DASHBOARD))    # utils 可 import
os.chdir(DASHBOARD)                    # st.Page("pages/...") 路徑正確
sys.argv[0] = str(APP_PATH)            # Streamlit 內部偵測正確

exec(compile(APP_PATH.read_text(encoding="utf-8"),
              str(APP_PATH), "exec"),
     {"__file__": str(APP_PATH), "__name__": "__main__"})
```

### 12.3 資源限制

| 項目 | Streamlit Cloud 免費級 | 本系統現況 |
|---|---|---|
| RAM | 1 GB | 所有頁面 `@st.cache_data` 下 ~600 MB |
| CPU | 1 vCPU | 初次載入 ~8 秒 |
| 磁碟 | 2 GB | 本 repo ~50 MB（排除大 parquet） |
| 連線 | 1,000 concurrent | 課堂用途足夠 |

**已知限制**：`outputs/feature_store_final.parquet` (272 MB) 超過 GitHub 100 MB 限制無法上傳。因此 `5_🗃️_Data_Explorer.py` 頁面在 Cloud 環境會顯示「feature store not found」——其他頁面透過 `outputs/reports/*.json` 正常運作。

## 14. 開發維運 SOP

### 14.1 新增一頁的 SOP

```python
# 步驟 1: 建立 程式碼/儀表板/pages/B_✨_新頁面.py
from pathlib import Path
import importlib.util, sys, streamlit as st

# 載入 utils
_dp = Path(__file__).resolve().parent.parent
spec = importlib.util.spec_from_file_location("utils", _dp / "utils.py")
_utils = importlib.util.module_from_spec(spec)
sys.modules["utils"] = _utils
spec.loader.exec_module(_utils)

# 注入全域設計系統
_utils.inject_custom_css()

# Hero 區塊（用統一組件）
st.markdown("""
<div class="gl-hero">
  <div class="gl-hero-eyebrow"><span class="gl-live">LIVE · 2026-04-20</span></div>
  <h1 class="gl-hero-title">新頁面標題</h1>
  <p class="gl-hero-sub">副標說明</p>
</div>
""", unsafe_allow_html=True)

# ... 你的內容 ...

# 步驟 2: 在 程式碼/儀表板/app.py 的 st.navigation dict 中註冊
newpage = st.Page("pages/B_✨_新頁面.py", title="新頁面", icon="✨")
pg = st.navigation({
    ...
    "🛡️ 模型治理與監控": [..., newpage],
})
```

### 14.2 更新設計 Token 的 SOP

1. **只在** `程式碼/儀表板/utils.py` 的 `THEME` dict 修改
2. 所有頁面透過 CSS 變數 `var(--gl-*)` 自動繼承新值
3. **不要** 在個別頁面寫死 hex 色碼，違反 single source of truth

### 14.3 新增一個 Phase 驗證的 SOP

```
1. 撰寫 程式碼/執行Phase{n}_{分析名}.py
   └─ 遵循既有 Phase 6 模式：
      - 讀取 feature_store_final
      - 計算指標
      - 輸出 outputs/reports/{分析名}_{horizon}.json
      - 輸出 outputs/figures/{分析名}_{horizon}.png

2. 在 程式碼/儀表板/pages/A_🔭_Phase6_深度驗證.py 或
   新建專屬頁面讀取 JSON → render

3. 更新本手冊 §8 或 §9 的對應章節

4. git commit + push → Streamlit Cloud 自動 redeploy
```

### 14.4 文案修訂流程

當發現頁面有過期數字（例如舊 ICIR 0.74-0.77）：

1. **grep 全局** 找出所有硬編碼位置
   ```bash
   grep -rn "0.74-0.77" 程式碼/
   ```
2. **分類處理**：
   - 當前生產環境用的 → 直接改為新數字
   - 歷史參考價值的（例如「Phase 1 初版」）→ 保留但加上警示橫幅（`<GlAlert variant="warn">`）
3. **交叉驗證** README / PPTX / docx 三處數字必須一致
